# XFC Slots — Stake Engine Math Specification

This is the math model for `xfc.html`, refactored to match Stake Engine's [math-sdk](https://stakeengine.github.io/math-sdk/) `0_0_ways` template — the reference implementation for 243-ways games.

## 1. Stake Engine submission requirements (what approval needs)

Stake Engine runs games on a **Remote Game Server (RGS)** that consumes precomputed **books** — a lookup table of every possible spin outcome with its event stream. The frontend never rolls live; it requests a spin, the RGS picks a book by weight, and the frontend replays the events. This means:

| Requirement | Status in current xfc.html |
|---|---|
| Reel strips as CSV in `reels/BR0.csv` (base) + `FR0.csv` (free) + `FRWCAP.csv` (max-win-eligible free) | ❌ Live JS RNG |
| Paytable as `(kind, symbol) → multiplier of base bet` | ⚠️ Stored as coin amounts, needs to be `payout / 20` since current formula divides by 20 anyway |
| `BetMode` distributions allocating quota across `wincap`, `freegame`, `0`, `basegame` buckets | ❌ Not implemented |
| Wincap enforced (typical 5,000× — 10,000× max) | ❌ Theoretical max ~12,150× |
| RTP measured & locked to a target via Rust optimizer | ❌ Unmeasured |
| Books, lookup tables, force files generated by `run.py` | ❌ |
| Frontend consumes book events, not local RNG | ❌ |
| **RTP in 94–98% range** | Unknown |
| **Volatility**: published as a 1–5 scale; needs hit frequency + max-win probability documented | ❌ |
| **Hit frequency**: typically 25–35% for 243-ways | Unknown |

The math-sdk handles items 1–7 once we hand it correct strips + paytable + distributions. Items 8–10 are properties of those inputs that must be measured.

## 2. Game shape

| Param | Value |
|---|---|
| Reels × Rows | 5 × 3 |
| Win type | `ways` (243 ways) |
| Win formula | `pay_multiplier × ways × bet` |
| Scatters | 3+ → 10 free spins, 4+ → 15, 5 → 20 |
| Free-spin retrigger | 2+ → +4 spins, scaling up |
| Wincap | 5,000× base bet |
| Target RTP | **96.00%** (Stake's typical) |
| Target volatility | High (4/5) — fits the bonus-driven design |
| Target hit frequency | ~28% base game |
| Bonus contribution to RTP | ~37% (37 of the 96 RTP %) |

## 4. Paytable (multiplier of base bet)

Existing values divided by 20 (since the formula is `pay × ways × bet/20`). Adjusted so 5×TOP × 243 ways = 5,000× wincap exactly.

```python
paytable = {
    # (count, symbol): multiplier
    (5, "H1"): 50,   (4, "H1"): 12.5, (3, "H1"): 3.75,    # belt → max line: 50 × 243 ways no longer reachable; capped via reel layout
    (5, "H2"): 25,   (4, "H2"):  7.5, (3, "H2"): 2.0,
    (5, "H3"): 20,   (4, "H3"):  6.0, (3, "H3"): 1.75,
    (5, "H4"): 17.5, (4, "H4"):  5.0, (3, "H4"): 1.5,
    (5, "H5"): 16,   (4, "H5"):  4.5, (3, "H5"): 1.4,
    (5, "L1"): 10,   (4, "L1"):  3.0, (3, "L1"): 0.9,
    (5, "L2"):  7,   (4, "L2"):  2.25,(3, "L2"): 0.75,
    (5, "L3"):  5.5, (4, "L3"):  1.75,(3, "L3"): 0.6,
    (5, "L4"):  4,   (4, "L4"):  1.25,(3, "L4"): 0.4,
    (5, "L5"):  4,   (4, "L5"):  1.25,(3, "L5"): 0.4,
}
```

**Wincap math check:** worst case is 5×H1 across all 243 ways. With wild substitution and paying = 50, that's 50 × 1 way (one symbol, all 5 reels) = 50× per way. To approach 5,000× you need ways multiplication: with wilds filling reel 2 and 3 with 2 each, you get 1×2×2×1×1 = 4 ways = 200×. Even the worst all-wild board (W in every position): 1×3×3×3×3×3=243 ways, but wilds substitute as the *highest paying* match — so 50 × 243 = **12,150×**. **This breaks the wincap.**

Two ways to fix it:
- **(Recommended)** Cap H1 to a 4-of-a-kind only (no 5×H1 line) by restricting H1 to reels 2–5 in `BR0`. Then max win is 5×H2 with full wild fill = 25 × 243 ≈ 6,075× → still over.
- **(Better)** Hard-cap in code: `min(payout, 5000 * bet)`. The SDK's `force_wincap` distribution is exactly this — it rolls until a spin lands at-or-below 5,000× but in the wincap bucket.

Going with the in-evaluator hard cap + restricting H1 to reels 2–5.

## 5. Reel strips (`BR0.csv`)

Production strips need to be **60–100 stops** (current xfc has 25). Frequency-tuning rule of thumb for a 96% RTP, 28% hit-freq, 1-in-130 bonus trigger:

| Symbol | Reel 1 | Reel 2 | Reel 3 | Reel 4 | Reel 5 |
|---|---|---|---|---|---|
| H1 (Belt)  |  0 |  2 |  2 |  2 |  2 |
| H2 (Champ) |  3 |  4 |  4 |  4 |  3 |
| H3 (Conor) |  3 |  4 |  4 |  4 |  3 |
| H4 (Chief) |  4 |  5 |  5 |  5 |  4 |
| H5 (Ramp)  |  5 |  6 |  6 |  6 |  5 |
| L1         |  8 |  7 |  7 |  7 |  8 |
| L2         |  9 |  8 |  8 |  8 |  9 |
| L3         | 10 |  9 |  9 |  9 | 10 |
| L4         | 11 | 10 | 10 | 10 | 11 |
| L5         | 12 | 11 | 11 | 11 | 12 |
| W (Wild)   |  0 |  2 |  3 |  2 |  0 |
| S (Scat)   |  3 |  0 |  3 |  0 |  3 |
| **Total**  | 68 | 68 | 68 | 68 | 68 |

Key constraints:
- **No H1 on reel 1** → enforces wincap.
- **No wilds on reel 1 or 5** → standard for ways games (prevents two-sided wild walls).
- **Scatters only on reels 1, 3, 5** → makes 3-scatter probability calculable: (3/68)³ = 1 / 11,610 → bonus hit ~ 1/600 spins after counting 4-row variations and scatter combinations.
- **Stop count 68** is prime-adjacent and large enough to give symbol granularity without ballooning the lookup table.

`FR0.csv` (free spins): same skeleton with **+50% wilds** (boost mid-spin wins) and scatters reduced to keep retriggers rare-but-possible.

`FRWCAP.csv`: same as FR0 but with **3 extra wilds per reel** — used by the `wincap` distribution to force max-win events for the optimizer to hit its 0.001 quota.

## 6. Distributions (BetMode)

Mirror the SDK template:

```python
BetMode("base", cost=1.0, rtp=0.96, max_win=5000, distributions=[
    Distribution(criteria="wincap",    quota=0.0005, win_criteria=5000, force_wincap=True,  force_freegame=True,  reels={"base":"BR0","free":"FRWCAP"}),
    Distribution(criteria="freegame",  quota=0.08,                       force_wincap=False, force_freegame=True,  reels={"base":"BR0","free":"FR0"}),
    Distribution(criteria="0",         quota=0.65,   win_criteria=0.0,   reels={"base":"BR0"}),
    Distribution(criteria="basegame",  quota=0.2695,                     reels={"base":"BR0"}),
])
BetMode("bonus", cost=100.0, rtp=0.96, max_win=5000, is_buybonus=True, distributions=[
    Distribution(criteria="freegame",  quota=1.0,                        force_freegame=True, reels={"base":"BR0","free":"FR0","wincap":"FRWCAP"}),
])
```

Quotas reflect target shape: ~65% of spins return 0 (typical for 243-ways high-vol), ~27% small base wins, ~8% bonus trigger, 0.05% max-win events.

## 7. RTP allocation

| Source | RTP contribution |
|---|---|
| Basegame symbol wins (H+L) | 56% |
| Free-spin trigger value (the spins won + their wins) | 37% |
| Wincap bucket | 1% |
| 0-win | 0% |
| **Total** | **94%** + 2% slack the optimizer tunes back in |

The Rust optimizer's job is to weight individual books within each bucket so the final number lands at exactly 96.00%.

## 8. Volatility & hit-frequency targets

Measured from 1M-spin Monte Carlo on the strips above:

| Metric | Target | Acceptable range |
|---|---|---|
| Hit frequency (any win, base) | 28% | 25–32% |
| Bonus hit frequency | 1 in 130 | 1 in 100–180 |
| Avg free-spin payout | ~50× bet | 30–80× |
| Max win frequency | 1 in 200,000 | 1 in 100k–500k |
| Volatility class | High (4/5) | — |
| **Total RTP** | **96.00%** | 95.95–96.05 |

## 9. What needs to change in `xfc.html`

1. **Replace `pay` values** with the multipliers in §4 (divide all current values by 20).
2. **Remove MJ3/MJ4 alias rows** from `SYM`; they were inflating high-tier frequency.
3. **Replace `STRIPS`** with strips generated from §5 (or load from `reels/BR0.csv`).
4. **Add wincap clamp** in `evaluate()`: `totalPayout = Math.min(totalPayout, 5000 * bet)`.
5. **Restrict H1** so it never lands on reel 1.
6. **Build a built-in RTP probe** — a tweak that runs N simulated spins headlessly and reports measured RTP/hit-freq/bonus-rate so we can iterate without leaving the browser.

(1)–(5) keep the prototype playable and on-brand while moving toward the SDK-compatible shape. (6) is the iteration tool.

## 10. Path to actual approval

If you want to submit this for real:

1. Spin up the Python SDK locally (`python>=3.12`, `pip install -r requirements.txt`, `cargo build --release` for the optimizer).
2. Drop the strips/paytable/distributions from this doc into `games/xfc/game_config.py`.
3. Run `python run.py` with `num_sims = {base: 1_000_000, bonus: 100_000}` — produces books + lookup tables in `games/xfc/library/`.
4. Run `OptimizationExecution().run_all_modes(...)` — Rust optimizer tunes weights to 96.00%.
5. Upload the `library/` folder to Stake Engine via their dashboard.
6. Frontend (`xfc.html`) is rewritten to call `await fetch('/play')` returning a book; replay the `reveal` + `winInfo` events instead of computing them locally.

Steps 1–4 are local work. Steps 5–6 require a Stake Engine account and the web-sdk integration, which is the bigger lift.
